# 需求澄清

在进入方案设计、任务拆解或代码实现前，先把原始需求整理成可执行输入。

## 适用时机

- 用户只给出一句话目标，例如“做一个权限系统”
- 需求里缺少角色、范围、约束、边界或成功标准
- 用户直接要方案，但原始描述明显过于模糊
- 需求中可能混有目标、实现手段和假设，尚未分离

## 核心目标

把“模糊诉求”转成“结构化需求”，并显式暴露不确定项，避免过早进入方案设计。

## 执行规则

1. 先复述用户真正想解决的问题，而不是直接给技术方案。
2. 优先补齐业务目标、使用对象、核心流程、边界条件和非目标范围。
3. 如果信息不足但不阻塞推进，先做合理假设，并把假设放进“待确认问题”。
4. 如果信息不足会直接影响方案方向，明确指出关键缺口。
5. 不要一次抛出大量开放式问题；优先归纳，再提最关键的少量确认点。

## 输出结构

默认按以下结构输出：

```md
需求目标：
用户场景：
核心流程：
边界条件：
异常情况：
非目标范围：
待确认问题：
```

详细模板见 [output-template.md](references/output-template.md)

## 字段说明

- `需求目标`：这件事最终要解决什么问题，成功结果是什么
- `用户场景`：谁在什么场景下使用，角色与触发时机是什么
- `核心流程`：主路径上的关键步骤，避免直接跳实现细节
- `边界条件`：权限、状态、规模、配置、数据范围、时序等约束
- `异常情况`：失败、空数据、冲突、重复提交、权限不足等情况
- `非目标范围`：本次不做什么，帮助控制范围蔓延
- `待确认问题`：真正影响后续方案判断的问题，数量尽量少且关键

## 输出要求

- 保持面向执行，不写空泛分析
- 可以补充合理推断，但要明确是推断
- 如果需求已经足够清晰，也仍然用该结构做一次收束
- 不直接进入技术选型，除非用户明确要求并且需求已清晰

## 与其他 workflow skill 的衔接

- 澄清完成后，再进入 `complexity-evaluation`
- 若用户直接要求方案设计，但当前输入明显模糊，优先先触发本 skill
