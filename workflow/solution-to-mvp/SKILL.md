# 方案到 MVP

这个 skill 负责把已经相对清晰的需求，转成可执行方案和 MVP 路线。

## 前置条件

优先基于以下输入执行：

- 已完成的需求澄清结果
- 已完成的复杂度评估结果

如果前两步没有显式给出，但当前上下文足够清晰，可以继续；否则应先回退到前置 skill。

## 负责内容

- 方案设计
- 主流方案对比
- 最终方案选择
- MVP 范围定义
- 后续迭代路线
- 方案确认后的文档沉淀询问

## 执行原则

1. 先说“为什么这样设计”，再说“怎么做”。
2. 对比方案时优先比较成本、复杂度、扩展性和风险，不做空泛罗列。
3. MVP 只保留最小闭环，不把理想状态全部塞进一期。
4. 迭代路线要和 MVP 明确分层，避免边界模糊。
5. 输出节奏必须服从复杂度等级。
6. 当用户确认方案后，先询问是否需要生成文档，而不是默认输出长文档。

## 按复杂度控制输出方式

详细模板见 [output-modes.md](references/output-modes.md)

### 简单需求

一次性输出：

```text
1. 需求理解
2. 初步方案
3. 最终方案
4. MVP
5. 迭代建议
```

### 中复杂需求

分两阶段输出：

```text
阶段 1：需求确认 + 方案设计
阶段 2：MVP + 迭代建议
```

### 高复杂需求

按以下步骤逐步推进：

```text
Step 1：需求确认
Step 2：主流方案对比
Step 3：最终方案
Step 4：MVP
Step 5：Roadmap
```

## 推荐输出结构

```md
需求理解：

候选方案对比：
- 方案 A：
- 方案 B：

最终方案：

MVP 范围：
- 必做
- 可延后

迭代路线：
- 第 1 阶段
- 第 2 阶段

风险与注意事项：
```

## 方案确认后的收口动作

当本轮已经完成讨论且用户确认方案方向后，追加一步：

```text
询问用户是否需要进一步生成：
1. MVP 文档
2. 迭代路线文档
```

执行要求：

- 不要在未确认前直接生成正式文档
- 文档产出是 `solution-to-mvp` 的确认后动作，不是前置动作
- 如果用户只需要口头方案，不强制生成文档
- 如果用户需要落文档，再按对应模板输出

文档模板见：

- [mvp-document-template.md](references/mvp-document-template.md)
- [iteration-document-template.md](references/iteration-document-template.md)

## 方案对比的最小维度

每次至少覆盖以下维度：

- 实现复杂度
- 对当前系统的改动范围
- 扩展性
- 交付速度
- 风险点

## MVP 定义规则

- 必须能形成一个最小可验证闭环
- 不为“未来可能用到”而提前做过多抽象
- 与正式完整方案相比，明确说明删减了什么
- 能落到功能清单、模块边界或里程碑层面

## 文档产出规则

### MVP 文档

用于沉淀“这一轮真正要做什么”。

内容应围绕：

- 目标
- 范围
- 核心流程
- 功能清单
- 非目标范围
- 验收标准
- 风险与约束

### 迭代路线文档

用于沉淀“在 MVP 之后准备怎么继续做”。

内容应围绕：

- 原始需求目标
- 当前已实现的 MVP 能力
- 尚未覆盖的能力
- 各改进项优先级
- 推荐迭代顺序
- 每阶段目标

## 迭代路线要求

- 以“先验证核心价值，再补完能力”为主线
- 建议按阶段说明每轮新增内容
- 如果存在高风险点，优先把风险前置验证

## 1-n 阶段的处理原则

MVP 完成后，不重新回到一整套复杂方案流程；优先基于以下输入生成迭代文档：

- 最开始的原始需求
- 当前已经实现的 MVP
- 当前缺口与待补能力
- 各项需求或改进项的优先级

目标是输出一份按优先级收敛后的迭代文档，而不是重新做一次从零开始的 MVP 设计。

## 与其他 workflow skill 的衔接

- 输入来自 `requirement-clarification` 与 `complexity-evaluation`
- 输出应能直接服务后续实现、拆任务或进入 `implementation` 类 skill
