# 复杂度评估

这个 skill 用来判断一个需求应该如何推进，而不是直接给出完整方案。

## 适用时机

- 已经有相对清晰的需求输入
- 需要判断是简单需求、中复杂需求还是高复杂需求
- 需要决定输出节奏，避免复杂需求一次性信息过载

## 评估维度

每项按 `0-2` 分评估，总分 `0-10`。

详细评分参考见 [scoring-matrix.md](references/scoring-matrix.md)

### 1. 业务复杂度

- `0`：单流程、规则简单
- `1`：多规则、多状态或多角色
- `2`：跨场景、跨角色、复杂业务链路

### 2. 技术复杂度

- `0`：普通页面或局部改动
- `1`：模块级联动或中等技术设计
- `2`：架构级改造、基础设施或高耦合系统改动

### 3. 不确定性

- `0`：需求明确、路径清晰
- `1`：部分信息模糊，需要少量假设
- `2`：探索性强、关键前提不稳定

### 4. 影响范围

- `0`：单模块或单页面
- `1`：多模块协同
- `2`：全局系统、平台级或多团队影响

### 5. 交付周期

- `0`：通常 1 天内可完成
- `1`：通常 1 周内可完成
- `2`：多周、多阶段或跨版本交付

## 等级判断

- `0-3`：简单需求
- `4-6`：中复杂需求
- `7-10`：高复杂需求

## 执行策略

- 简单需求：一次性完整输出
- 中复杂需求：两阶段输出
- 高复杂需求：必须 step-by-step

详细策略见 [execution-strategy.md](references/execution-strategy.md)

## 输出结构

```md
复杂度评分：X/10
等级：
评分依据：
- 业务复杂度：X
- 技术复杂度：X
- 不确定性：X
- 影响范围：X
- 交付周期：X
建议执行方式：
说明：
```

## 输出要求

1. 每个维度都要给出简短依据，不只给分数。
2. 如果前置需求仍然模糊，明确指出“当前评分基于现有信息的初步判断”。
3. 评分的目标是决定推进方式，不是追求绝对精确。
4. 结论要能直接衔接 `solution-to-mvp` 的输出节奏。

## 与其他 workflow skill 的衔接

- 默认在 `requirement-clarification` 之后使用
- 评估完成后，把“等级 + 建议执行方式”传给 `solution-to-mvp`
