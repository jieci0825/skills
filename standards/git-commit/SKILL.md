name: git-commit
description: 根据当前改动生成符合 Conventional Commits 规范的中文 Git 提交信息，可附带详细变更说明
---

请检查当前 Git 暂存区或工作区改动，并生成一份符合规范的 Git 提交信息。

## 输出目标

优先生成包含标题和详细说明的提交内容，格式如下：

type(scope): description

- 变更点 1
- 变更点 2
- 变更点 3

如果本次改动非常简单，也可以只输出标题，不强制补充详细说明。

---

## 标题格式要求

标题必须严格使用以下格式：

type(scope): description

例如：

feat(登录): 新增短信验证码登录
fix(消息列表): 修复重复渲染问题
refactor(请求封装): 重构接口错误处理逻辑

---

## 允许的 type

仅允许使用以下前缀：

- feat
- fix
- refactor
- style
- docs
- test
- build
- perf
- ci
- chore
- revert

---

## scope 要求

- scope 必须使用中文
- 根据本次改动自动判断最准确的范围
- 应优先概括核心模块、页面、组件、功能域或基础设施层
- 避免使用空泛范围，例如：
  - 项目
  - 系统
  - 模块
  - 功能
- 若改动涉及多个区域，优先选择影响最大的核心范围
- scope 尽量简短明确

---

## description 要求

- 必须使用中文
- 简洁明确
- 以动词开头
- 不要加句号
- 聚焦“本次主要做了什么”
- 尽量控制在 10 到 20 个字以内

---

## 详细说明要求

当改动包含多个明确变更点时，在标题下补充详细说明：

- 每一行使用 `- ` 开头
- 必须使用中文
- 每条描述一个具体改动点
- 聚焦实际改了什么，不写空话
- 避免重复标题内容
- 一般输出 2 到 5 条
- 不要写“其他优化”“若干调整”这类模糊表述

例如：

feat(用户中心): 新增头像上传功能

- 增加头像选择与预览交互
- 接入头像上传接口
- 补充上传失败提示逻辑

---

## 类型判断规则

根据改动内容选择最合适的 type：

- 新功能使用 feat
- 修复缺陷使用 fix
- 代码重构使用 refactor
- 性能优化使用 perf
- 样式调整使用 style
- 文档更新使用 docs
- 测试相关使用 test
- 构建工具或依赖调整使用 build
- CI 流程调整使用 ci
- 杂项维护使用 chore
- 回滚提交使用 revert

如果同时存在多类改动，优先选择最核心的一类。

---

## 禁止事项

不要建议提交以下内容：

- node_modules
- dist
- build 产物

---

## 输出要求

- 默认输出“标题 + 详细说明”
- 如果改动非常单一，可以只输出标题
- 不要添加解释
- 不要添加代码块
- 不要输出多个候选项
- 输出内容必须可直接作为 Git 提交信息使用
