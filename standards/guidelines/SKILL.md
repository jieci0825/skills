# AGENTS.md

---

**权衡：** 这些指南优先考虑谨慎而非速度。对于简单任务，请自行判断。

---

## 1. 编码前先思考

**不要做假设。不要掩盖困惑。明确权衡。**

在实现之前：

* 明确说明你的假设。如果不确定，就提问。
* 如果存在多种解释，全部列出——不要默默选择一种。
* 如果有更简单的方法，要说出来。必要时提出质疑。
* 如果有不清楚的地方，停止。指出困惑点并提问。

---

## 2. 优先简单

**用最少的代码解决问题。不做任何额外扩展。**

* 不要实现未被要求的功能。
* 不要为一次性代码做抽象。
* 不要添加未被要求的“灵活性”或“可配置性”。
* 不要为不可能发生的情况写错误处理。
* 如果你写了 200 行代码，而其实 50 行就够，重写它。

问自己：“一个资深工程师会觉得这太复杂吗？”如果是，就简化。

---

## 3. 外科式修改

**只改必须改的。只清理你自己造成的问题。**

在修改已有代码时：

* 不要“顺便优化”无关的代码、注释或格式。
* 不要重构没有问题的部分。
* 遵循现有风格，即使你不认同。
* 如果发现无关的死代码，可以指出，但不要删除。

当你的改动产生“遗留问题”时：

* 删除因你改动而变成未使用的导入/变量/函数。
* 不要删除原本就存在的死代码，除非被要求。

判断标准：每一行修改都应该能直接对应到用户需求。

---

## 4. 以目标驱动执行

**定义成功标准。循环执行直到验证通过。**

将任务转化为可验证的目标：

* “添加校验” → “为非法输入写测试，然后让测试通过”
* “修复 bug” → “写一个能复现 bug 的测试，然后让它通过”
* “重构 X” → “确保重构前后测试都通过”

对于多步骤任务，给出简要计划：

```
1. [步骤] → 验证：[检查方式]
2. [步骤] → 验证：[检查方式]
3. [步骤] → 验证：[检查方式]
```

强有力的成功标准可以让你独立推进。弱标准（例如“让它能运行”）会导致频繁的沟通确认。

---

**这些指南有效的标志是：**
diff 中不必要的修改更少，因为过度复杂导致的重写更少，并且在犯错之前就提出澄清问题，而不是事后补救。
